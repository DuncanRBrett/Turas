## Overview

The segmentation module produces a self-contained, interactive HTML report alongside the standard Excel outputs. The report is designed for presentation-ready delivery and stakeholder review.

**Why HTML?**

- **Self-contained** -- A single `.html` file with no external dependencies. Open it in any modern browser.
- **Interactive** -- Sticky navigation bar, pinnable views, and slide-export capabilities let analysts curate findings for presentations without leaving the browser.
- **Brand-customisable** -- Colours, titles, and section visibility are all controlled through the config Excel file.
- **SVG charts** -- All charts are rendered as inline SVG, so they scale cleanly at any zoom level and export as crisp PNGs.

**Two report variants are generated depending on the analysis mode:**

| Mode | Trigger | Report Type |
|------|---------|-------------|
| **Final** | `k_fixed` is set in config | Full segmentation report with profiles, validation, cards, and rules |
| **Exploration** | `k_fixed` is blank | K-selection report with elbow plot, silhouette chart, metrics table, and solution previews |

The entry point is `generate_segment_html_report()` in `lib/html_report/99_html_report_main.R`, which routes to the appropriate builder based on `results$mode`.

## Configuration

All HTML report settings are specified in the Config sheet of the segmentation configuration Excel file alongside other analysis parameters.

### Enabling the Report

#### html_report

- **Purpose:** Enable or disable HTML report generation
- **Required:** No
- **Data Type:** TRUE/FALSE
- **Default:** `TRUE`
- **Notes:** Set to FALSE to skip HTML output entirely and only produce Excel files

### Brand Customisation

#### brand_colour

- **Purpose:** Primary brand colour used for headers, nav bar, chart bars, and section titles
- **Required:** No
- **Data Type:** Hex colour string
- **Default:** `#323367`
- **Example:** `#1A5276`
- **Notes:** Feeds the CSS variable `--seg-brand`

#### accent_colour

- **Purpose:** Secondary colour used for highlights, recommended-k markers, and warning indicators
- **Required:** No
- **Data Type:** Hex colour string
- **Default:** `#CC9900`
- **Example:** `#E67E22`
- **Notes:** Feeds the CSS variable `--seg-accent`

#### report_title

- **Purpose:** Title displayed in the report header and browser tab
- **Required:** No
- **Data Type:** Text
- **Default:** Value of `project_name`, or `"Segmentation Analysis"` if neither is set
- **Example:** `Customer Segmentation Q4 2025`

### Section Visibility Flags

Each major section of the HTML report can be toggled on or off via a config flag. All flags default to TRUE except `html_show_rules`, which defaults to FALSE because classification rules require the enhanced features pipeline.

| Parameter | Default | Section Controlled |
|-----------|---------|-------------------|
| `html_show_exec_summary` | TRUE | Executive Summary |
| `html_show_overview` | TRUE | Segment Overview (sizes chart + composition table) |
| `html_show_validation` | TRUE | Cluster Validation (silhouette chart + metrics table) |
| `html_show_importance` | TRUE | Variable Importance (eta-squared bar chart) |
| `html_show_profiles` | TRUE | Segment Profiles (heatmap + profile table) |
| `html_show_rules` | FALSE | Classification Rules (requires enhanced features) |
| `html_show_cards` | TRUE | Segment Action Cards |
| `html_show_guide` | TRUE | Interpretation Guide |

**Notes:**

- A section will also be hidden if its underlying data is not available, regardless of the flag. For example, `html_show_importance = TRUE` has no effect if ANOVA-based variable importance was not computed during the analysis.
- GMM Membership is shown automatically when the clustering method is `gmm` or `mclust` and membership probability data is present. There is no separate flag.
- Pinned Views is always shown; it is a functional workspace area, not an analytical section.

### Complete Configuration Example

```
Parameter               | Value
------------------------|--------------------------------
html_report             | TRUE
brand_colour            | #323367
accent_colour           | #CC9900
report_title            | Customer Segmentation 2025
html_show_exec_summary  | TRUE
html_show_overview      | TRUE
html_show_validation    | TRUE
html_show_importance    | TRUE
html_show_profiles      | TRUE
html_show_rules         | FALSE
html_show_cards         | TRUE
html_show_guide         | TRUE
```

## Report Sections (Final Mode)

The final-mode HTML report is composed of the sections described below, rendered in order from top to bottom.

### Header

The header is a gradient banner displaying:

- Turas logo and module name
- Report title (from `report_title` or `project_name` config)
- Prepared-by line with analyst name and timestamp
- Inline badges showing method, k, sample size, and average silhouette score

The header always appears and is not toggleable.

### Executive Summary

**Config flag:** `html_show_exec_summary`

A high-level narrative overview of the segmentation solution. Content is auto-generated from the analysis results and includes:

- **Quality banner** -- Colour-coded assessment of solution quality (Excellent, Good, Moderate, Limited) based on the average silhouette score
- **Key findings** -- Bullet points summarising the number of segments, dominant segment, smallest segment, and top differentiating variables
- **Key insights** -- Contextual observations derived from the segment profile data

The executive summary is intended for senior stakeholders who need a quick read before diving into the detail sections.

### Segment Overview

**Config flag:** `html_show_overview`

Two complementary views of segment composition:

1. **Segment Sizes Chart (SVG)** -- Horizontal bar chart showing each segment sorted by size (descending). Each bar displays the segment name, count (n), and percentage. Bar opacity fades by rank to create visual hierarchy.

2. **Composition Table** -- Tabular view of the same data with columns for Segment ID, Name, Size (n), Size (%), and a Status badge. Segments below 5% of the sample are flagged with a "Small" warning badge.

### Cluster Validation

**Config flag:** `html_show_validation`

Statistical quality metrics for the clustering solution:

1. **Silhouette Chart (SVG)** -- Horizontal bar chart showing the average silhouette score per cluster. Bars are colour-coded by quality:
   - Green (>= 0.50): strong cohesion and separation
   - Brand colour (0.25--0.49): acceptable structure
   - Amber (< 0.25): weak structure

   A dashed reference line is drawn at 0.25 to mark the minimum acceptable threshold.

2. **Validation Metrics Table** -- Rows for each metric with Value and Interpretation columns. Metrics include:

   | Metric | Description |
   |--------|-------------|
   | Average Silhouette | Overall cluster cohesion/separation (range -1 to 1) |
   | Between/Total SS | Proportion of variance explained by the clustering |
   | Method | Clustering algorithm used (kmeans, hclust, gmm) |
   | Linkage | Hierarchical clustering linkage method (hclust only) |
   | Cophenetic Correlation | Dendrogram fit quality (hclust only) |
   | BIC | Bayesian Information Criterion (GMM only) |

### Variable Importance

**Config flag:** `html_show_importance`

Identifies which clustering variables most differentiate the segments:

1. **Importance Bar Chart (SVG)** -- Horizontal bar chart sorted by importance (descending). The display metric is chosen in this priority order:
   - Normalised importance percentage (eta-squared / total eta-squared * 100)
   - Raw eta-squared values
   - F-statistic values

   Bars fade in opacity by rank to visually distinguish top drivers from lower-ranked variables. Variable labels are resolved through the question labels file when available.

2. **Underlying Data** -- The chart is backed by the variable importance data frame extracted from the ANOVA-based profiling step, which includes:
   - Variable name
   - Eta-squared (effect size)
   - F-statistic
   - Rank
   - Importance percentage

### Segment Profiles

**Config flag:** `html_show_profiles`

The core analytical section showing how segments differ across all clustering variables:

1. **Heatmap Chart (SVG)** -- A colour-coded grid with variables as rows and segments as columns. Each cell displays the cluster centroid mean value. Cell background colour is interpolated based on z-scores computed across clusters for each variable:
   - Blue shading indicates below-average values (negative z-scores)
   - White indicates near-average values
   - Red shading indicates above-average values (positive z-scores)

   Text colour automatically switches to white when the background is too dark for legibility (|z| > 2.0).

2. **Profile Table** -- Full data table with one row per clustering variable and columns for:
   - Variable name (resolved to label when a question labels file is provided)
   - One column per segment showing the segment mean
   - Overall mean
   - F-statistic (when available)
   - Eta-squared (when available)

   Segment mean cells are colour-tinted based on the index score (segment mean / overall mean * 100):
   - Index > 120: strong blue highlight
   - Index 105--120: light blue highlight
   - Index 80--95: light amber highlight
   - Index < 80: light red highlight

### Classification Rules

**Config flag:** `html_show_rules` (default: FALSE)

Decision-tree classification rules generated by the enhanced features pipeline. The section displays:

- **Rules Table** -- One row per rule showing:
  - Rule text (in monospace `<code>` format, e.g. `IF Q02 > 6.5 AND Q05 <= 3.2 THEN Advocates`)
  - Predicted segment name
  - Per-rule accuracy percentage

- **Footer row** -- Overall classification accuracy across all segments

This section only appears when `html_show_rules = TRUE` **and** the enhanced features pipeline has produced classification rules.

### Segment Cards

**Config flag:** `html_show_cards`

Executive-ready action cards displayed in a responsive grid (auto-fit, minimum 280px per card). Each card includes:

- **Segment name** -- Displayed as the card heading in brand colour
- **Segment size** -- Count and percentage
- **Profile** -- What defines this segment (top distinguishing variables)
- **Strengths** -- Highest-scoring variables relative to the overall mean
- **Pain points** -- Lowest-scoring variables relative to the overall mean
- **Recommended actions** -- Suggested next steps based on the segment profile

Cards are generated by `generate_segment_cards()` in `lib/segment_cards.R`.

### GMM Membership

**Shown automatically when:** method is `gmm` or `mclust` and membership probabilities are present

A table summarising soft membership assignment quality for Gaussian Mixture Models:

| Column | Description |
|--------|-------------|
| Segment | Segment name |
| Mean Probability | Average assignment probability for observations assigned to this segment |
| Max Uncertainty | Highest uncertainty (1 - min probability) among assigned observations |
| N Borderline | Count of observations whose maximum probability is below 0.60 |

Borderline counts greater than zero are displayed as warning badges.

### Interpretation Guide

**Config flag:** `html_show_guide`

A reference section explaining how to read the report, including:

- Silhouette score interpretation thresholds
- Index score colour coding explained
- Effect size (eta-squared) benchmarks
- Tips for evaluating segment quality

This section is static content generated by `build_seg_guide_section()` in the page builder.

### Pinned Views

**Always shown**

A workspace area at the bottom of the report for curating findings. See [Interactive Features](#interactive-features) for details on how pinning works.

## Exploration Mode Report

When `k_fixed` is left blank, the module runs in exploration mode and produces a different HTML report focused on helping the analyst choose the optimal number of clusters.

**Meta tag:** `turas-report-type = "segment-exploration"`

### Report Structure

The exploration report has a simpler structure than the final report:

1. **Header** -- Gradient banner showing report title, method, k range tested, number of successful solutions, and generation timestamp.

2. **Recommendation Card** -- A highlighted panel (amber border, warm background) displaying:
   - Recommended k value
   - Rationale text explaining why this k was chosen
   - Confidence score (when available)
   - Bulleted list of supporting reasons

3. **Cluster Evaluation Charts** -- Two charts displayed side-by-side in a responsive grid:

   - **Elbow Plot (SVG)** -- Line chart showing within-cluster sum of squares (Within-SS) on the y-axis against k on the x-axis. The recommended k is highlighted with an accent-coloured marker and "recommended" label. Includes gridlines, axis labels, and a plot-area border.

   - **Silhouette by k Chart (SVG)** -- Line chart showing average silhouette score by k. Each data point displays its value. The recommended k is highlighted with a ring marker and value label.

4. **Clustering Metrics Table** -- Comparison table showing, for each k tested:
   - Average silhouette score
   - Within-cluster sum of squares
   - Calinski-Harabasz (CH) index (when available)
   - Davies-Bouldin (DB) index (when available)
   - Between-SS / Total-SS ratio (when available)

   The recommended k row is highlighted in the accent colour and marked with an asterisk.

5. **Solution Previews** -- One card per k value showing:
   - k value heading (highlighted for recommended k)
   - "RECOMMENDED" badge (for the chosen k)
   - Silhouette score badge (colour-coded: green >= 0.5, amber >= 0.25, red < 0.25)
   - Mini horizontal bar chart of segment sizes with count and percentage labels

6. **Footer** -- Standard Turas footer with generation details.

### Exploration-Specific CSS

The exploration report adds a few supplementary styles on top of the shared `build_seg_css()` stylesheet:

- `.seg-exploration-grid` -- Two-column grid for the chart panel, collapsing to single column below 900px
- `.seg-chart-panel` -- White card container for individual charts
- `.seg-recommendation-card` -- Highlighted card with amber border and warm background
- `.seg-recommendation-title`, `.seg-recommendation-text`, `.seg-recommendation-reason` -- Typography for the recommendation card

## Design System

### CSS Namespace

All CSS classes in the segment report use the `seg-` prefix to prevent collisions with other Turas module reports when loaded in the Report Hub. Examples:

```
seg-body, seg-header, seg-section, seg-table, seg-th, seg-td,
seg-chart, seg-badge, seg-callout, seg-action-card, seg-pin-btn
```

### Design Tokens (CSS Variables)

The report declares CSS custom properties at `:root` scope:

| Variable | Default | Purpose |
|----------|---------|---------|
| `--seg-brand` | `#323367` | Primary brand colour (overridden by `brand_colour` config) |
| `--seg-accent` | `#CC9900` | Accent colour (overridden by `accent_colour` config) |
| `--seg-text` | `#1e293b` | Primary text colour |
| `--seg-text-muted` | `#64748b` | Secondary text colour |
| `--seg-text-faint` | `#94a3b8` | Tertiary / hint text colour |
| `--seg-bg` | `#f8f7f5` | Page background (warm off-white) |
| `--seg-card` | `#ffffff` | Card / section background |
| `--seg-border` | `#e2e8f0` | Border colour |
| `--seg-success` | `#059669` | Positive indicators |
| `--seg-warning` | `#F59E0B` | Warning indicators |
| `--seg-danger` | `#c0392b` | Negative / error indicators |

### Brand Customisation

The `build_seg_css()` function accepts `brand_colour` and `accent_colour` parameters and injects them into the CSS template via string replacement of `BRAND_COLOUR` and `ACCENT_COLOUR` placeholders. This means every instance of the brand colour throughout the report -- headers, nav links, section titles, chart bars, card borders -- updates automatically from a single config value.

### Typography

The report uses the system font stack:

```css
font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif;
```

Key typographic settings:

| Element | Size | Weight |
|---------|------|--------|
| Body text | 13px | 400 |
| Table header | 11px | 600, uppercase |
| Section title | 16px | 700 |
| Report title | 28px | 700 |
| Badge text | 10px | 600, uppercase |
| Chart labels | 11px | 400--500 |

### Table Styling

Tables use the `.seg-table` class with:

- Full-width layout, collapsed borders
- Alternating row backgrounds (even rows: `#f9fafb`)
- Row hover highlight (`#f8fafc`)
- Header row with light purple tint (`#f0f1f8`)
- Heatmap cell tinting via utility classes:
  - `.seg-td-high` (blue), `.seg-td-mod-high` (light blue)
  - `.seg-td-mod-low` (amber), `.seg-td-low` (red)

### Section Layout

Each content section is wrapped in a `.seg-section` card:

```css
.seg-section {
  background: white;
  border: 1px solid #e2e8f0;
  border-radius: 8px;
  padding: 24px 28px;
  margin-bottom: 20px;
}
```

Content is constrained to 1100px maximum width, centred on the page.

## Interactive Features

### Section Navigation

A sticky horizontal navigation bar sits below the header, remaining visible as the user scrolls. Each visible section appears as a nav link. The active section is underlined with the brand colour.

Navigation is powered by `seg_navigation.js`, which uses `IntersectionObserver` to track which section is currently in view and updates the active nav link accordingly.

Clicking a nav link scrolls smoothly to the corresponding section.

### Pinned Views

The Pinned Views system allows analysts to collect specific charts and tables from anywhere in the report into a single workspace for export or printing.

**How to pin a view:**

1. Hover over any chart or table wrapper to reveal the pin button (a small icon in the top-right corner)
2. Click the pin button to add the component to the Pinned Views section
3. The pin button turns solid (active state) to indicate it has been pinned
4. Scroll to the Pinned Views section at the bottom to see collected items

**Pinned Views panel actions:**

| Button | Action |
|--------|--------|
| Add Section | Adds a custom text section to the pinned area |
| Export All as PNG | Exports each pinned view as an individual PNG image |
| Print / PDF | Opens the browser print dialog with pinned views formatted for clean printing |
| Clear All | Removes all pinned views from the workspace |

Pinned views data is stored in a hidden `<script type="application/json">` element with id `seg-pinned-views-data`. This state persists within the page session but is not saved to disk.

The pinned views system is implemented in `seg_pinned_views.js`.

### Slide Export (PNG)

Individual sections and pinned views can be exported as PNG images for inclusion in presentations. The export uses the browser's native canvas rendering via `seg_slide_export.js`.

**How it works:**

1. The export function captures the target DOM element
2. Renders it to a canvas at high resolution
3. Converts to PNG and triggers a browser download

This functionality requires no external CDN or library -- it is self-contained within the report JavaScript.

### Insight Editors

Each section includes a collapsible "Add insight" area where analysts can type contextual notes directly in the report. These notes are stored in a hidden `<textarea>` element (`seg-insight-store`) and are included when the section is pinned or exported.

## Integration with Report Hub

The HTML report includes metadata tags in the `<head>` section that allow the Turas Report Hub to identify and catalogue it:

```html
<meta name="turas-report-type" content="segment">
<meta name="turas-module-version" content="11.0">
<meta name="turas-source-filename" content="seg_final_report.html">
```

For exploration reports, the report type is set to `segment-exploration`.

These meta tags enable the Report Hub to:

- Recognise the report as a Turas module output
- Display it with the correct module icon and label
- Group it with other reports from the same analysis run
- Apply the appropriate viewer settings

The `seg-` CSS namespace ensures that segment report styles do not conflict with reports from other Turas modules (e.g., `kd-` for keydriver, `cat-` for catdriver) when multiple reports are displayed in the hub.

## Troubleshooting

### htmltools Package Not Installed

**Error:**
```
REFUSED: PKG_HTMLTOOLS_MISSING
Package 'htmltools' is required for HTML report generation but is not installed.
```

**Fix:**
```r
install.packages("htmltools")
```

The `htmltools` package is the only additional dependency required for HTML report generation. It is checked at the start of the pipeline by `validate_segment_html_inputs()` in `00_html_guard.R`.

### Empty or Missing Sections

**Symptom:** A section flag is set to TRUE in the config but the section does not appear in the report.

**Cause:** The section requires data that was not produced by the analysis. Section visibility is controlled by both the config flag **and** the presence of underlying data.

**Common cases:**

| Section | Required Data | Why It May Be Missing |
|---------|--------------|----------------------|
| Variable Importance | `variable_importance` (eta-squared from ANOVA) | Profiling step did not run or failed |
| Segment Profiles | `profile_data` (clustering profile) | Profiling step did not run |
| Classification Rules | `enhanced$classification_rules` | Enhanced features not enabled or `html_show_rules = FALSE` |
| Segment Cards | `enhanced$segment_cards` | Enhanced features not enabled |
| GMM Membership | `gmm_membership` (probability matrix) | Method is not GMM, or GMM did not return probabilities |

**Diagnosis:** Check the R console output during analysis. Each step logs its progress and any warnings. Look for messages like:

```
    Built 3 tables
    Built 4 charts
```

A lower count than expected indicates a build failure for that component. Warnings are reported at the end of the pipeline.

### HTML File Is Empty or Very Small

**Symptom:** The output `.html` file exists but is 0 bytes or much smaller than expected.

**Possible causes:**

1. **Page assembly failed** -- Check the console for `CALC_PAGE_BUILD_FAILED`. This usually indicates a data transformation error.
2. **Write failed mid-stream** -- The writer uses atomic write (write to `.tmp` then rename). If the temp file exists but the final file does not, the rename failed.
3. **Disk space** -- Verify available disk space in the output directory.

**Diagnosis:**
```r
# Check if the temp file was left behind
file.exists(paste0(output_path, ".tmp"))

# Check file size
file.info(output_path)$size
```

### Charts Not Rendering

**Symptom:** A section appears but the SVG chart area is blank.

**Cause:** The chart builder returned an empty SVG string because the input data was NULL or had zero rows.

**Fix:** Verify that the upstream analysis produced the expected data. For example:

- Sizes chart requires `segment_sizes` data frame with at least one row
- Silhouette chart requires `sil_per_cluster` (named vector, data frame, or list)
- Importance chart requires `variable_importance` data frame with `eta_squared`, `f_statistic`, or `importance_pct` column
- Heatmap chart requires `centers` matrix from the cluster result

### Brand Colours Not Applying

**Symptom:** The report uses default colours despite setting `brand_colour` in the config.

**Common causes:**

1. **Typo in parameter name** -- Must be exactly `brand_colour` (British spelling). `brand_color` will be ignored.
2. **Invalid hex format** -- Must include the `#` prefix (e.g. `#323367`, not `323367`).
3. **Config not parsed** -- Verify the config Excel file has the correct sheet name and the parameter appears in the parameter column.

### Report Hub Does Not Recognise the File

**Symptom:** The Report Hub does not display the segment report or shows it as an unknown file type.

**Fix:** Verify the `<head>` section of the HTML file contains the required meta tags:

```html
<meta name="turas-report-type" content="segment">
```

If the tags are missing, the page builder may have encountered an error before the meta tags were injected. Re-run the analysis and check for errors.

### Output Directory Does Not Exist

**Error:**
```
REFUSED: IO_CANNOT_CREATE_DIR
Cannot create output directory: /path/to/output
```

**Fix:** Ensure the parent directory exists and you have write permissions. The writer will attempt to create the output directory recursively, but cannot create paths where the parent itself does not exist or where permissions are restricted.

## Pipeline Architecture

For developer reference, the HTML report generation follows a 6-step pipeline:

```
99_html_report_main.R        Main orchestrator / entry point
  |
  +-- 00_html_guard.R        Step 1: Validate inputs (htmltools, results, config, path)
  +-- 01_data_transformer.R  Step 2: Transform results into flat html_data structure
  +-- 02_table_builder.R     Step 3: Build HTML tables (overview, profile, validation, etc.)
  +-- 05_chart_builder.R     Step 4: Build SVG charts (sizes, silhouette, importance, heatmap)
  +-- 03_page_builder.R      Step 5: Assemble CSS + HTML sections + JS into full page
  +-- 04_html_writer.R       Step 6: Atomic write to disk (.tmp -> rename)
  +-- 06_exploration_report.R  Alternative path for exploration mode
```

JavaScript files in `lib/html_report/js/`:

| File | Purpose |
|------|---------|
| `seg_utils.js` | Shared utility functions |
| `seg_navigation.js` | Sticky nav bar and section tracking |
| `seg_pinned_views.js` | Pin, collect, and manage views |
| `seg_slide_export.js` | PNG export for sections and pinned views |

All JavaScript is inlined into the HTML at build time, making the output fully self-contained.
