# Turas Hub App — Development Specification

**Version:** 1.0 Draft **Date:** 22 March 2026 **Author:** Duncan Brett
/ The Research LampPost (Pty) Ltd **Status:** Specification — not yet in
development

------------------------------------------------------------------------

## 1. Executive Summary

The Turas Hub App is a lightweight local web application that provides a
working environment for analysts to browse, annotate, and curate
findings across multiple Turas HTML reports. It complements the existing
single-file Report Hub (which remains the client deliverable and
archival format) by removing the constraints of embedding everything in
one HTML file.

**Key distinction:** - **Single-file hub** = deliverable you send to
clients, archive to OneDrive, email to stakeholders - **Hub App** = your
working environment where you analyse, pin, annotate, compare, and
export

------------------------------------------------------------------------

## 2. Problem Statement

The single-file Report Hub works well for 5-10 reports but has inherent
limitations:

1.  **No live CSS fidelity on pins** — pinned tables/charts lose their
    original report styling because they render outside the report's CSS
    context
2.  **File size grows with pins** — every pinned SVG chart and styled
    table adds to the HTML file
3.  **No cross-project search** — each hub file is self-contained; you
    cannot search findings across projects
4.  **No PowerPoint export** — pins export as PNG only; analysts need
    slides for client presentations
5.  **Memory pressure** — all report HTML is base64-encoded in the DOM
    simultaneously
6.  **No collaboration** — the file is a snapshot; two people cannot
    work on annotations concurrently

The Hub App solves all of these by loading reports directly from the
filesystem.

------------------------------------------------------------------------

## 3. Target Users

| User | Role | Key Needs |
|-------------------|-------------------|----------------------------------|
| **Duncan** | Lead analyst, developer | Full analytical workflow, cross-project search, PowerPoint export |
| **Jess** | Project operator | Simple GUI, reliable pins, clear error messages, export to client deliverables |
| **Future analysts** | Team members | Browse reports, add insights, build presentation decks |

**Technical context:** Users run R/RStudio. Reports are generated by
Turas modules and stored in project folders on local drives or OneDrive.
Users are comfortable with a browser-based interface but are not
developers.

## 4. Architecture

### 4.1 High-Level Architecture

```         
┌─────────────────────────────────────────────────────────────┐
│                    Browser (localhost:port)                  │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
│  │ Project  │  │ Report   │  │  Pin     │  │  Export    │  │
│  │ Browser  │  │ Viewer   │  │  Board   │  │  Manager  │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────────┘  │
│                                                             │
│  ┌─────────────────────────────────────────────────────────┐│
│  │              Shared State (IndexedDB)                   ││
│  │   pins[] · annotations[] · project_index · preferences  ││
│  └─────────────────────────────────────────────────────────┘│
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP (localhost)
┌──────────────────────────┴──────────────────────────────────┐
│                     R/Plumber API Server                     │
│                                                             │
│  GET /projects              List project folders            │
│  GET /projects/:id/reports  List HTML reports in a project  │
│  GET /report?path=...       Serve a report HTML file        │
│  POST /export/pptx          Generate PowerPoint from pins   │
│  POST /export/hub           Generate single-file hub        │
│  GET /search?q=...          Full-text search across reports │
│                                                             │
│  ┌─────────────────────────────────────────────────────────┐│
│  │              File System Access                         ││
│  │   ~/Projects/**/*.html  · config files · pin JSON       ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Technology Stack

| Layer | Technology | Rationale |
|------------------|---------------------------|---------------------------|
| **Server** | R + Plumber | Already in the R ecosystem; analysts have R installed; Plumber is lightweight HTTP |
| **Frontend** | Vanilla HTML/CSS/JS | Consistent with existing Turas report architecture; no build step; no framework churn |
| **State** | IndexedDB (browser) + JSON sidecar files | Pins persist across sessions; JSON files back up to OneDrive |
| **Export** | officer (R package) | PowerPoint generation from R; mature, well-tested |
| **Reports** | iframe (direct <file://>) or proxied via server | Full CSS fidelity — report loads exactly as standalone |

### 4.3 Alternative: Electron App

If browser-only proves limiting (e.g., <file://> iframe restrictions),
consider wrapping in Electron: - Same frontend code - Direct filesystem
access without a server - Distributable as a single .app / .exe -
**Trade-off:** Larger distribution size (\~150MB), update management

**Recommendation:** Start with R/Plumber. Move to Electron only if
cross-origin iframe issues block progress.

## 5. Feature Specification

### 5.1 Project Browser

**Purpose:** Navigate between projects and their reports.

**Requirements:** - F1. Scan a configurable root directory (or list of
directories) for project folders - F2. Each project folder is expected
to contain one or more `.html` report files - F3. Display projects as
cards with: project name, report count, last modified date, total size -
F4. Support OneDrive, local drives, and network paths - F5. Allow
adding/removing project root directories in preferences - F6. Show a
search bar to filter projects by name - F7. Remember the last-opened
project on next launch

**Project detection heuristic:** - Any folder containing at least one
`.html` file with a `<meta name="turas-report-type">` tag - OR any
folder containing a `*_Report_Hub_Config.xlsx` file

### 5.2 Report Viewer

**Purpose:** View individual reports with full fidelity, exactly as they
appear standalone.

**Requirements:** - F8. Load reports in iframes served via the Plumber
API (avoids <file://> CORS issues) - F9. Reports render identically to
standalone — same CSS, same JS, same interactivity - F10. Tab bar across
the top for switching between reports (same UX as single-file hub) -
F11. Hide report headers (hub provides its own) — inject bridge CSS via
postMessage or DOM API - F12. Make report nav strips sticky within the
iframe - F13. Support up to 15 reports per project (same as single-file
hub edge case) - F14. Lazy-load iframes on first tab activation - F15.
Unload iframes when more than 5 are loaded (LRU eviction) to manage
memory

### 5.3 Pin Board

**Purpose:** Curate findings from across reports into a
presentation-ready collection.

**Requirements:** - F16. Central pin board accessible as a tab (same
position as single-file hub) - F17. Pins captured from reports via the
same pin mechanism (MutationObserver on pinned-views-data store) - F18.
**Full CSS fidelity on pinned tables** — render table HTML inside a
hidden iframe with the source report's CSS, then snapshot as a
high-resolution PNG (canvas.toDataURL at 3x) - F19. Pinned charts render
with full styling (SVG with resolved CSS variables) - F20. Each pin
shows: source report badge, title, subtitle, insight text (editable),
chart, table - F21. Pins are reorderable via drag-and-drop (not just
up/down buttons) - F22. Section dividers for narrative structure (same
as single-file hub) - F23. Pin data stored in IndexedDB AND backed up as
`{project}/.turas_pins.json` sidecar file - F24. Pins persist across
browser sessions and app restarts - F25. Pin count badge on the Pin
Board tab

**Pin capture flow (full fidelity):**

```         
1. User clicks "Pin" in report (inside iframe)
2. Report's savePinnedData() fires → updates pinned-views-data store
3. Hub MutationObserver detects change
4. Hub reads new pin data (chartSvg, tableHtml, insightText)
5. For tableHtml: create hidden iframe loading source report CSS,
   render table inside it, capture as PNG at 3x via canvas
6. Store: { chartSvg, tableHtml, tablePng, insightText, ... }
7. Display: show tablePng for on-screen quality;
   use tableHtml for PowerPoint export (officer can render HTML tables)
```

### 5.4 Insight & Annotation System

**Purpose:** Add analyst commentary to pinned views.

**Requirements:** - F26. Each pin has an editable insight area (markdown
→ rendered HTML, same as single-file hub) - F27. Double-click to edit,
blur to save (same UX) - F28. Insights auto-save to IndexedDB on every
keystroke (debounced 500ms) - F29. Support markdown: **bold**, *italic*,
\## headings, \> blockquotes, - bullets - F30. Per-project executive
summary and background text areas (overview page) - F31. All text edits
saved to the `{project}/.turas_annotations.json` sidecar file

### 5.5 Export Manager

**Purpose:** Export pins and insights to client-ready formats.

**Requirements:** - F32. **Export to PowerPoint (.pptx)** — one slide
per pin, using the `officer` R package - Title slide with project name,
date, branding - Each pin → one slide: title, chart image (PNG at 3x),
table (native PPTX table), insight text - Section dividers → section
header slides - Branded template with configurable colours and logo -
F33. **Export to PDF** — browser print with `@media print` styles (same
as single-file hub) - F34. **Export individual pin as PNG** — same
SVG-native approach as single-file hub (3x resolution) - F35. **Export
all pins as PNGs** — sequential download or ZIP - F36. **Generate
single-file hub** — call existing `combine_reports()` R function to
produce the archival deliverable from the current project - F37. Export
progress indicator for PowerPoint (may take a few seconds with many
pins)

### 5.6 Cross-Project Search

**Purpose:** Find findings across all projects.

**Requirements:** - F38. Full-text search across pin titles, insights,
report metadata - F39. Search results show: project name, report name,
pin title, matching snippet - F40. Click a result to open the project
and navigate to the relevant pin/report - F41. Search index built on
first scan, updated incrementally when projects change - F42. Search
covers: pin text, annotation text, report titles, question codes

### 5.7 Preferences

**Requirements:** - F43. Configurable project root directories (list of
paths to scan) - F44. Default brand colour and accent colour - F45.
Default logo path for exports - F46. Auto-save interval for
annotations - F47. Theme: light (default) — dark mode as future
enhancement - F48. Stored in `~/.turas/hub_app_config.json`

## 6. Data Model

### 6.1 Pin Object

``` json
{
  "id": "pin-1711100000-abc12",
  "type": "pin",
  "source": "tracker",
  "sourceLabel": "Brand Health Tracker",
  "title": "Q05 - Overall Satisfaction",
  "subtitle": "Top-2 box score by segment",
  "insightText": "## Key Finding\n\nSatisfaction dropped **3pp** among Urban segment...",
  "chartSvg": "<svg ...>...</svg>",
  "tableHtml": "<table ...>...</table>",
  "tablePng": "data:image/png;base64,...",
  "pngDataUrl": "data:image/png;base64,...",
  "pinMode": "all",
  "timestamp": 1711100000000,
  "projectId": "brand-health-2026",
  "reportPath": "/path/to/tracker_report.html"
}
```

### 6.2 Project Index

``` json
{
  "id": "brand-health-2026",
  "name": "Brand Health Study 2026",
  "path": "/Users/duncan/Projects/BrandHealth2026/",
  "reports": [
    {
      "path": "Tracker_Report.html",
      "label": "Brand Tracker",
      "type": "tracker",
      "size": 2450000,
      "lastModified": "2026-03-20T14:30:00Z"
    }
  ],
  "lastOpened": "2026-03-22T09:15:00Z",
  "pinCount": 8
}
```

### 6.3 Sidecar Files

Each project folder gets two hidden sidecar files:

```         
ProjectFolder/
├── Report_A.html
├── Report_B.html
├── .turas_pins.json          ← pin data (array of pin objects)
└── .turas_annotations.json   ← executive summary, background, notes
```

These files: - Are JSON, human-readable, git-friendly - Sync via
OneDrive alongside the reports - Can be imported into the single-file
hub via `combine_reports()` - Are the source of truth (IndexedDB is a
cache for performance)

``` r
#* @post /api/export/hub
function(project_id, req) { ... }

# ── Search ──

#* Search across all projects
#* @param q Search query
#* @get /api/search
function(q) { ... }

# ── Static Files ──

#* Serve the Hub App frontend
#* @assets /path/to/hub_app/static /
list()
```

### 7.2 Security Considerations

-   Server binds to `127.0.0.1` only (localhost) — no network access
-   No authentication needed (local use only)
-   File paths validated against configured root directories (prevent
    directory traversal)
-   Report HTML served read-only (no write access via API)

## 9. Launch Mechanism

``` r
source("modules/report_hub/app/launch_hub_app.R")
launch_hub_app(
  project_dirs = c(
    "~/Projects",
    "~/OneDrive/DB Files/Projects"
  ),
  port = 7860,
  open_browser = TRUE
)
```

### 9.2 From Turas Shiny GUI

Add a "Hub App" button to the existing `launch_turas.R` Shiny interface.
Clicking it: 1. Starts the Plumber server in a background R process 2.
Opens the browser to `http://localhost:7860` 3. Shiny GUI shows "Hub App
running on port 7860" with a "Stop" button

### 9.3 From Command Line

``` bash
Rscript -e 'source("modules/report_hub/app/launch_hub_app.R"); launch_hub_app()'
```

## 10. Compatibility with Single-File Hub

The Hub App and single-file hub are complementary. Data flows between
them:

```         
Hub App                          Single-File Hub
────────                         ───────────────
.turas_pins.json    ──export──►  combine_reports()  ──►  Combined.html
.turas_annotations.json ──►      (embeds pins as     (self-contained
                                  hub-pinned-data)     deliverable)

Combined.html       ──import──►  Hub App reads pins
(saved with edits)               from hub-pinned-data
                                 and populates pin board
```

**Import:** When opening a project that contains a single-file hub
(`*_Combined_*.html`), the app can extract existing pins from the
`hub-pinned-data` element and populate the pin board.

**Export:** The "Generate Hub File" button calls `combine_reports()`
with the current project's reports and injects the pin board data into
the output.

## 11. Development Phases

### Phase 1: Core Viewer (2-3 days)

-   Plumber server with project listing and report serving
-   Frontend shell: project browser, report tabs, iframe loading
-   Bridge injection (header hiding, sticky navs)
-   Launch from R console

### Phase 2: Pin Board (2-3 days)

-   Pin forwarding from reports (MutationObserver + monkey-patching)
-   Pin board UI with cards, section dividers, reordering
-   IndexedDB persistence + JSON sidecar backup
-   Insight editing with markdown

### Phase 3: Export (2 days)

-   PNG export (existing SVG-native approach)
-   PowerPoint export via `officer`
-   Generate single-file hub from project

### Phase 4: Polish (1-2 days)

-   Cross-project search
-   Preferences panel
-   Import from existing single-file hubs
-   Error handling, loading states, edge cases

**Total estimated effort: 7-10 days**

## 14. Risks and Mitigations

| Risk | Impact | Mitigation |
|------------------|----------------------|--------------------------------|
| iframe CORS with <file://> URLs | Reports won't load | Serve via Plumber proxy (planned architecture) |
| Plumber port conflict | App won't start | Auto-detect available port; configurable |
| OneDrive file locking | Sidecar writes fail | Retry with backoff; warn user |
| Large report files (\>10MB) | Slow iframe load | Loading indicator; lazy-load only active tab |
| Browser IndexedDB quota | Pins lost | JSON sidecar files as source of truth; IndexedDB is cache |
| officer PPTX rendering limits | Complex tables may not render | Fall back to PNG image of table in slide |

## 15. Out of Scope (v1.0)

-   Multi-user collaboration (real-time co-editing)
-   Cloud deployment (stays local only)
-   Dark mode
-   Mobile responsive design
-   Version history for annotations
-   AI-assisted insight generation
-   Direct database connectivity
-   Custom report templates

These may be considered for future versions based on usage patterns.

------------------------------------------------------------------------

## 16. Success Criteria

The Hub App is considered production-ready when:

1.  An analyst can launch it from R, open any project, and browse all
    reports
2.  Pins from every Turas module type forward correctly to the pin board
3.  Pinned tables render with full visual fidelity (matching standalone
    reports)
4.  Insights are editable, persist across sessions, and sync to sidecar
    files
5.  PowerPoint export produces a client-ready deck with charts, tables,
    and insights
6.  The app works reliably on Chrome, Edge, Safari, and Firefox
7.  Performance is responsive with 15 reports and 50+ pins
8.  Cross-project search returns relevant results within 1 second
9.  The "Generate Hub File" button produces a valid single-file hub with
    all pins included
10. Jess can use it without developer assistance
