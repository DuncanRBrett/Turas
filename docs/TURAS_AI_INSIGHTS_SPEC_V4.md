## Overview

Add AI-generated insight callouts to Turas Tabs reports, clearly labelled as AI-assisted observations and visually distinct from researcher-written commentary.

### The two-tier content model

The report supports two complementary types of per-question commentary:

1. **AI callouts** ŌĆö observational pattern-spotting generated by the LLM. These flag notable differences, highlight where subgroups over- or under-index, and surface data patterns worth attention. Clearly labelled as AI-assisted. Distinct visual treatment (different background, AI icon). Integrated with pin-to-slide for selective inclusion in findings decks.

2. **Researcher commentary** ŌĆö strategic interpretation written by the researcher (Duncan, Jess, or the white-label partner). This is the "here's what this means for your business" layer. Uses the existing report styling with no AI labelling. Written manually in the config, never AI-generated.

These are not competing tiers. They are complementary voices: the AI says "here are things in the data you should notice," the researcher says "here's what matters and what to do about it." The client understands the difference intuitively because the roles are different.

### Executive summary

A synthesised narrative (3-5 paragraphs) rendered at the top of the report. Its visual treatment depends on the delivery context:

- **Researcher-reviewed delivery** (TRL projects, analytics-as-a-service where Duncan or Jess reviews the output): AI-drafted, researcher-edited, presented as "Key findings ┬Ę Reviewed by research team." No AI labelling ŌĆö because the researcher has genuinely reviewed and shaped it.
- **Unreviewed delivery** (white-label partners running Turas independently, self-service clients): AI-generated without researcher review. Labelled honestly as "Key findings ┬Ę AI-assisted" with the same visual treatment as AI callouts (distinct background, AI icon). The content is the same; the labelling reflects whether a human checked it.

This is controlled by a single config flag: `config$ai_insights$exec_summary_reviewed`. When TRUE, standard report styling. When FALSE, AI callout styling. The default is TRUE for TRL projects and FALSE for white-label.

### Why label AI content

Transparency protects trust. A SAMRA-accredited researcher with a 30-year reputation has more to lose from undisclosed AI text than from honest labelling. If an AI callout says something slightly off, it's an AI observation that didn't quite land ŌĆö not the researcher's professional opinion being wrong. The labelling provides an honest buffer while the researcher retains full control via review and pin-to-slide selection.

The same principle extends to the executive summary in white-label contexts. If no human has reviewed it, it should not claim human review. The label must be honest about what actually happened, not aspirational about what should have happened.

### Quality passes

Because report generation is a batch process with no real-time constraint, the pipeline includes multi-pass quality assurance. Each AI callout is verified against source data before being accepted. The full set of callouts is then reviewed for selectivity ŌĆö removing any that merely restate what a chart already shows without adding interpretation. The executive summary is generated in two stages: structured pattern identification first, then narrative generation from those patterns. These passes add seconds, not minutes, and substantially raise the quality floor.

## Success Criteria

These are measurable objectives. The feature is working when all are met.

### Must-have

1. **Selectivity:** AI callouts generate for 35-60% of questions in a typical report. Below 35% suggests the model is too conservative. Above 60% suggests it is commenting on unremarkable data. Every question getting a callout is a failure.
2. **Factual accuracy:** Zero hallucinated numbers. Every number cited in every callout matches the source data exactly. Verified by the automated verification pass.
3. **Significance discipline:** No callout uses assertive language ("significantly higher") for a difference that is not flagged as statistically significant in the source data.
4. **Generation time:** Full pipeline (all callouts + verification + selectivity + executive summary) completes in under 5 minutes for a 27-question report. Speed is not critical, but it should not block the workflow.
5. **Graceful degradation:** Report generates successfully with zero AI-related errors when the API is unavailable, the key is missing, or the model returns malformed output.
6. **Idempotency:** Manually edited callouts, researcher commentary, and pin state are preserved across regeneration.
7. **Observational scope:** No AI callout contains strategic recommendations ("the company should..."). The callouts describe what the data shows, not what to do about it.

### Nice-to-have

8. **Executive summary quality:** Judged "usable with minor edits" by Duncan on 4 out of 5 test runs against real client datasets.
9. **Workflow efficiency:** A researcher can review, edit, and pin all AI callouts in under 10 minutes for a standard 25-question report.
10. **Client validation:** At least one trusted client (CCPB, Martin, or equivalent) confirms the two-tier labelling feels honest and the callouts add value.

## Data Extraction

### Function: `extract_insights_data(tabs_output, config)`

Extracts a structured list from the Turas tabs analytical output. This is what gets sent to the LLM ŌĆö never raw data, only aggregated outputs.

#### Study context

```r
extract_study_context <- function(tabs_output, config) {
  list(
    report_title  = config$report_title,
    total_n       = config$total_n,
    weighted      = config$weighted,
    fieldwork     = config$fieldwork,
    banner_groups = names(config$banners),
    banner_points = lapply(config$banners, function(b) {
      list(labels = b$labels, base_sizes = b$base_sizes)
    })
  )
}
```

#### Per-question extraction

```r
extract_question_data <- function(q) {
  list(
    q_code          = q$code,
    q_title         = q$title,
    q_type          = q$type,
    chart_type      = q$chart_data$chart_type,
    response_labels = q$chart_data$labels,
    results         = q$chart_data$columns,
    priority_metric = if (!is.null(q$chart_data$priority_metric)) {
      list(
        label  = q$chart_data$priority_metric$label,
        values = q$chart_data$priority_metric$values
      )
    },
    significance    = extract_sig_flags(q),
    net_rows        = q$net_rows,
    base_sizes      = q$base_sizes
  )
}
```

#### Significance extraction ŌĆö boolean format

Convert Turas's internal column-letter significance notation into plain-language boolean flags:

```r
# Example output for Q001 NPS, Segment banner:
# list(
#   list(
#     measure     = "Promoter (9-10)",
#     column      = "Premium",
#     value       = 60,
#     direction   = "higher",
#     vs_columns  = c("Standard", "Budget", "New Customer"),
#     significant = TRUE
#   )
# )
```

The extraction function reads from Turas's internal R-side sig test results. Column-letter codes are translated to display labels before the data reaches the LLM.

#### Context window management

For the executive summary, the combined payload of all questions may approach context window limits on smaller models. Implement `extract_question_data_compact()` which returns a reduced payload per question ŌĆö topline total scores, priority metric values, and significant differences only (dropping full per-category breakdowns). Use the compact version for the executive summary if the full payload exceeds 80% of the model's context window. Estimate token count as `nchar(jsonlite::toJSON(payload)) / 4`.

#### What NOT to extract

- No individual respondent records
- No raw response vectors
- No demographic variables beyond the banner definitions
- No weighting variables or design metadata
- No open-ended response text
- No PII of any kind

## Prompt Construction

### Function: `build_insight_prompt(data, study_context, prompt_type)`

Prompt types: `"ai_callout"`, `"verification"`, `"selectivity"`, `"exec_patterns"`, `"exec_narrative"`.

### System prompt (AI callout ŌĆö shared across all callout calls)

Store as text constants in `R/ai_prompts.R`.

```
You are a data analyst assistant supporting a senior market
researcher. Your role is OBSERVATIONAL: you spot patterns,
flag notable differences, and surface things in the data worth
attention. You do NOT provide strategic recommendations or tell
the client what to do ŌĆö that is the researcher's role.

ANALYTICAL STANDARDS
- Only reference numerical values explicitly present in the data
  payload. Do not infer, calculate, or fabricate any numbers.
- Only use assertive language ("significantly higher") when the
  significance flag is TRUE. For non-significant differences, use
  hedged language ("numerically higher, though not statistically
  significant") or omit.
- If any base sizes are below 50, set confidence to "medium" and
  note in data_limitations. Below 30, set confidence to "low"
  and do not draw quantitative conclusions.
- If the data shows no clear pattern, set has_insight to FALSE.
  Do not force commentary on unremarkable data.

WRITING STANDARDS
- Write in clear, direct prose. No bullet points. No headers.
- Use specific numbers: "scored 72%, significantly above 58%"
  not "scored well above average".
- Describe what the data shows, not what it means strategically.
  "Eastern Cape scores 23% good/excellent on delivery, compared
  to 66% in Gauteng" is your job. "The company needs to
  investigate its Eastern Cape logistics" is the researcher's job.
- Do not use: "delve", "dive into", "unpack", "landscape",
  "nuanced", "robust" (unless statistical), "leverage",
  "holistic", "key takeaway", "it is worth noting".
```

```

identify any callouts that should be removed because they

A callout should be REMOVED if it:
- Merely restates what the chart already shows without adding

- States something trivially obvious from the data
- Repeats the same observation made in another callout
- Is generic enough to apply to any dataset

A callout should be KEPT if it:
- Highlights a non-obvious pattern or relationship
- Notes a significant difference worth attention
- Identifies a surprising finding or counter-intuitive result
- Provides context that makes the data more meaningful

Return the q_codes of callouts to remove.

{callouts_json}
```

## LLM Calls via ellmer

### Function: `create_chat(config)`

```r
create_chat <- function(config) {
  ai <- config$ai_insights

  switch(ai$provider,
    "anthropic" = ellmer::chat_anthropic(
      model       = ai$model,
      api_key     = Sys.getenv(ai$api_key_env)
    ),
    "openai" = ellmer::chat_openai(
      model       = ai$model,
      api_key     = Sys.getenv(ai$api_key_env)
    ),
    "google" = ellmer::chat_google_gemini(
      model       = ai$model,
      api_key     = Sys.getenv(ai$api_key_env)
    ),
    "ollama" = ellmer::chat_ollama(
      model       = ai$ollama_model,
      base_url    = ai$ollama_url
    ),
    stop("Unknown AI provider: ", ai$provider)
  )
}
```

### Function: `call_insight_model(prompt, schema, config)`

```r
call_insight_model <- function(prompt, schema, config) {
  tryCatch({
    chat <- create_chat(config)
    chat$set_system_prompt(prompt$system)

    result <- chat$chat_structured(
      prompt$user,
      type = schema,
      temperature = config$ai_insights$temperature,
      max_tokens  = config$ai_insights$max_tokens
    )

    result
  }, error = function(e) {
    warning("AI insight generation failed: ", e$message)
    NULL
  })
}
```

### Note on ellmer API

Check `ellmer` CRAN docs at build time for exact method signatures. The method may be `$chat_structured()` or `$extract_data()` depending on version. The core pattern is the same: pass a type schema, get a typed list back.

## Verification Pass

### Function: `verify_callout(callout, question_data, config)`

After generating each callout, send the narrative and source data to a verification call. This catches hallucinated numbers ŌĆö the single highest-risk failure mode.

```r
verify_callout <- function(callout, question_data, config) {
  if (!isTRUE(config$ai_insights$verify_callouts)) return(callout)
  if (!isTRUE(callout$has_insight)) return(callout)

  prompt <- build_insight_prompt(
    data = list(narrative = callout$narrative, question_data = question_data),
    study_context = NULL,
    prompt_type = "verification"
  )

  result <- call_insight_model(prompt, verification_schema, config)

  if (is.null(result)) {
    callout$verified <- FALSE
    return(callout)
  }

  if (isTRUE(result$numbers_accurate) && isTRUE(result$significance_accurate)) {
    callout$verified <- TRUE
    return(callout)
  }

  # Verification failed ŌĆö return details for regeneration
  callout$verified <- FALSE
  callout$verification_issues <- result$mismatches
  callout
}
```

### Regeneration on verification failure

In the orchestration function, if verification fails, regenerate the callout with the verification feedback included in the prompt:

```r
# In generate_all_insights(), after initial generation:
if (!isTRUE(callout$verified)) {
  for (attempt in seq_len(config$ai_insights$max_verification_attempts)) {
    # Regenerate with verification feedback
    prompt$user <- paste0(
      prompt$user,
      "\n\nPREVIOUS ATTEMPT HAD ERRORS:\n",
      callout$verification_issues,
      "\n\nPlease correct these specific issues."
    )
    callout <- call_insight_model(prompt, ai_callout_schema, config)
    if (!is.null(callout)) {
      callout <- verify_callout(callout, q_data, config)
      if (isTRUE(callout$verified)) break
    }
  }
  # If still not verified after max attempts, suppress the callout
  if (!isTRUE(callout$verified)) {
    callout$has_insight <- FALSE
    warning(sprintf("Callout for %s failed verification after %d attempts ŌĆö suppressed",
                    q_code, config$ai_insights$max_verification_attempts))
  }
}
```

## Selectivity Pass

### Function: `rank_callouts(config)`

After all callouts are generated and verified, review the full set for quality. This removes callouts that are technically accurate but add no interpretive value.

```r
rank_callouts <- function(config) {
  if (!isTRUE(config$ai_insights$rank_callouts)) return(config)

  # Collect all callouts with has_insight = TRUE
  callout_summaries <- list()
  for (q_code in names(config$questions)) {
    ai <- config$questions[[q_code]]$ai_callout
    if (!is.null(ai) && isTRUE(ai$has_insight)) {
      callout_summaries[[q_code]] <- list(
        q_code    = q_code,
        q_title   = config$questions[[q_code]]$title,
        narrative = ai$narrative
      )
    }
  }

  if (length(callout_summaries) < 3) return(config)  # Too few to rank

  prompt <- build_insight_prompt(
    data = callout_summaries,
    study_context = NULL,
    prompt_type = "selectivity"
  )

  result <- call_insight_model(prompt, selectivity_schema, config)

  if (!is.null(result) && length(result$remove_q_codes) > 0) {
    for (q_code in result$remove_q_codes) {
      if (q_code %in% names(config$questions)) {
        config$questions[[q_code]]$ai_callout$has_insight <- FALSE
        message(sprintf("Selectivity pass removed callout for %s: %s",
                        q_code, result$reasoning))
      }
    }
  }

  config
}
```

## Two-Stage Executive Summary

### Function: `generate_executive_summary(all_q_data, study_context, config)`

Stage 1 identifies structured patterns. Stage 2 writes the narrative from those patterns. Two calls produce better output than one because the model reasons about what matters before committing to prose.

```r
generate_executive_summary <- function(all_q_data, study_context, config) {
  # Check context window ŌĆö use compact extraction if payload is large
  payload_json <- jsonlite::toJSON(all_q_data, auto_unbox = TRUE)
  estimated_tokens <- nchar(payload_json) / 4
  if (estimated_tokens > 80000) {
    all_q_data <- lapply(all_q_data, extract_question_data_compact)
    payload_json <- jsonlite::toJSON(all_q_data, auto_unbox = TRUE)
  }

  # Stage 1: Identify patterns
  prompt_patterns <- build_insight_prompt(
    all_q_data, study_context, "exec_patterns"
  )
  patterns <- call_insight_model(prompt_patterns, exec_patterns_schema, config)

  if (is.null(patterns)) return(NULL)

  # Stage 2: Write narrative from patterns
  prompt_narrative <- build_insight_prompt(
    data = list(patterns = patterns, all_q_data = all_q_data),
    study_context = study_context,
    prompt_type = "exec_narrative"
  )

  exec_config <- modifyList(config, list(
    ai_insights = modifyList(config$ai_insights, list(
      max_tokens = config$ai_insights$exec_summary_max_tokens
    ))
  ))

  result <- call_insight_model(prompt_narrative, insight_executive_schema, exec_config)
  result
}
```

## Orchestration

### Function: `generate_all_insights(tabs_output, config)`

Main entry point. Called once during report generation.

```r
generate_all_insights <- function(tabs_output, config) {
  if (!isTRUE(config$ai_insights$enabled)) return(config)

  study_context <- extract_study_context(tabs_output, config)

  # === Per-question AI callouts ===
  if (isTRUE(config$ai_insights$generate_per_question)) {
    for (q_code in names(config$questions)) {
      q <- config$questions[[q_code]]

      # Skip if excluded or already has a callout
      if (isTRUE(q$ai_callout_exclude)) next
      if (!is.null(q$ai_callout) && !is.null(q$ai_callout$has_insight)) next

      q_data <- extract_question_data(q)
      prompt <- build_insight_prompt(q_data, study_context, "ai_callout")

      # Generate
      callout <- call_insight_model(prompt, ai_callout_schema, config)
      if (is.null(callout)) next

      # Verify (with regeneration on failure)
      if (isTRUE(callout$has_insight)) {
        callout <- verify_callout(callout, q_data, config)

        if (!isTRUE(callout$verified)) {
          for (attempt in seq_len(config$ai_insights$max_verification_attempts)) {
            regen_prompt <- prompt
            regen_prompt$user <- paste0(
              prompt$user,
              "\n\nPREVIOUS ATTEMPT HAD ERRORS:\n",
              callout$verification_issues,
              "\n\nPlease correct these specific issues."
            )
            callout <- call_insight_model(regen_prompt, ai_callout_schema, config)
            if (is.null(callout)) break
            callout <- verify_callout(callout, q_data, config)
            if (isTRUE(callout$verified)) break
          }
          if (!isTRUE(callout$verified)) {
            callout$has_insight <- FALSE
            warning(sprintf("Callout for %s suppressed ŌĆö failed verification", q_code))
          }
        }
      }

      callout$pinned <- FALSE
      config$questions[[q_code]]$ai_callout <- callout

      Sys.sleep(0.5)  # Rate limiting courtesy
    }

    # Selectivity pass across all callouts
    config <- rank_callouts(config)
  }

  # === Executive summary ===
  if (isTRUE(config$ai_insights$generate_exec_summary)) {
    existing <- config$ai_insights$executive_summary
    if (is.null(existing) || nchar(existing$narrative %||% "") == 0) {
      all_q_data <- lapply(config$questions, extract_question_data)
      result <- generate_executive_summary(all_q_data, study_context, config)
      if (!is.null(result)) {
        config$ai_insights$executive_summary <- result
      }
    }
  }

  config
}
```

### Idempotency

- Manual edits to callout text are preserved across regeneration
- Researcher commentary is never touched by any AI function
- Pin state is preserved across regeneration
- Manually setting `has_insight = FALSE` suppresses a callout without triggering regeneration
- To regenerate a specific callout: `config$questions$QXXX$ai_callout <- NULL`
- To regenerate all: `clear_all_callouts(config)` which NULLs all `ai_callout` fields
- `clear_all_callouts()` never touches `researcher_commentary`

## Error handling

| Scenario | Behaviour |
|---|---|
| API key not set | Warning logged, callouts skipped, report generates normally |
| API call timeout | Warning logged, that question skipped, continues to next |
| API returns error | Warning logged, that question skipped, continues to next |
| Ollama not running | Warning logged, callouts skipped, report generates normally |
| Structured output parse failure | Warning logged, that question skipped, continues to next |
| Verification fails after max attempts | Callout suppressed (has_insight set FALSE), warning logged |
| Selectivity pass fails | All callouts retained (fail-open), warning logged |
| Executive summary Stage 1 fails | No executive summary generated, warning logged |
| `has_insight` is FALSE | No callout panel for that question (expected behaviour) |
| All API calls fail | Report generates normally with no callout panels |

The report must never fail to generate because of AI errors. The AI layer is additive only. All quality passes fail open ŌĆö if they error, the content passes through rather than being blocked.

## Testing

### Automated verification (built into pipeline)

Every callout is verified against source data before acceptance. This is not a test you run separately ŌĆö it runs as part of report generation.

### Manual verification process

1. Run `generate_all_insights()` on the Demo_CX_Crosstabs dataset
2. Review each generated AI callout against the actual data
3. Confirm: every number cited in a callout matches the data
4. Confirm: every significance claim matches the sig flags
5. Confirm: `has_insight` is FALSE for flat/unremarkable questions
6. Confirm: selectivity pass removed at least some low-value callouts
7. Confirm: callouts are observational, not strategic (no "the company should...")
8. Confirm: confidence is "medium"/"low" where base sizes warrant it
9. Confirm: AI callouts are visually distinct from researcher commentary
10. Confirm: pin-to-slide toggle works (pinned callouts print, unpinned don't)
11. Confirm: client-side toggle shows/hides all AI callouts
12. Confirm: executive summary renders with correct styling per `exec_summary_reviewed`
13. Confirm: report generates successfully when API is unavailable
14. Confirm: manually edited callouts and researcher commentary are preserved on re-run
15. Confirm: manually suppressed callouts (has_insight set FALSE) are not regenerated

### Real-data testing (before production use)

Test against at least three real past client datasets ŌĆö not just the demo data. Real data has messy variable labels, inconsistent scales, small subgroups, ambiguous significance patterns, and questions where the "right" interpretation is genuinely debatable. Prompt refinement should happen against real data, not demo data.

Budget a full day of prompt tuning per module. The prompts in this spec are a starting point. The difference between a callout that restates the chart and one that adds genuine interpretive value is in the prompt detail, and that detail emerges from iteration against real outputs.

### Jess workflow testing

Before considering the feature production-ready, have Jess run a complete report workflow: generate callouts, review them, pin the useful ones, add researcher commentary where needed, export the deck. Her feedback on whether the callouts save time or create review overhead is the real acceptance test.

### Expected metrics

For a 27-question report like the Demo_CX dataset:
- 10-16 callouts generated initially (before selectivity pass)
- 8-14 retained after selectivity (2-4 removed as low-value)
- 0 callouts suppressed by verification (if prompts are well-tuned)
- Full pipeline completes in 2-4 minutes
- Researcher review takes 5-10 minutes

## Notes for implementation

- Read `duncan-coding-standards` skill before writing any code
- The `data-chart-data` JSON attributes in existing Turas HTML contain the chart data structure ŌĆö the extraction functions should work from the R-side data objects that produce this JSON, not parse the HTML
- Significance in Turas tables uses column-letter notation internally. The extraction function must translate these to plain-language boolean flags before the data reaches the LLM
- The AI callout system prompt scopes the model as an observational assistant, NOT a senior analyst. Strategic interpretation belongs in researcher commentary only. Do not merge the roles
- The pin-to-slide functionality should integrate with Turas's existing pin/export mechanism if one exists, or establish the pattern if it doesn't
- The `Ō£”` sparkle character is used as the AI icon. If Inter does not render it cleanly, substitute with an inline SVG sparkle icon
- The `­¤ōī` pin icon uses an emoji. Consider replacing with inline SVG for consistent cross-platform rendering
- Store prompts as character constants in `R/ai_prompts.R` so they can be refined without touching function code
- Pin the `ellmer` version in `renv` ŌĆö the package is actively developed and method signatures may change between releases. Test before upgrading
- `ellmer` supports batch processing via `parallel_chat_structured()` ŌĆö could speed up per-question generation but is not required for MVP given speed is not a constraint
- For white-label deployments, set `exec_summary_reviewed = FALSE` in the white-label config template
- Consider showing the prototype (or a report with AI callouts) to one trusted client before shipping broadly. Their reaction to the two-tier labelling and the callout quality is the best validation available
