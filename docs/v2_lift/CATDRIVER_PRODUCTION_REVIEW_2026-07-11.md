## 0. Executive verdict

CatDriver's core regression machinery is better than its siblings' — the coefficient-to-level mapper is genuinely rigorous, refusal discipline in the engine is real, and unweighted binary/ordinal runs produce defensible importance and odds ratios. But the module is **not solid**: weighted multinomial runs ship mathematically invalid importance rankings under PASS (negative chi-squares); the entire subgroup-comparison feature is dead on arrival on every real run (column-contract mismatch, error swallowed, PASS); and the module's refusal system is broken module-wide — every user-fixable config mistake presents as "BUG_INTERNAL_ERROR: report to the Turas development team" with the actionable fix text destroyed. The shipped template cannot load, the GUI reports success on failures, the documented CLI entry point doesn't work, and the flagship multi-outcome (unified) report opens on a blank Overview tab and loses user-authored slides on save.

The suite's 659 passes are real but structurally green: the subgroup-comparison tests never construct the data shape production produces, and the tests source files in the opposite order to the GUI, so they bind the *correct* refusal signature that production never sees.

**Order of work: fix correctness + the refusal system first (Phase 1), then config/GUI/template/report quick fixes (Phase 2), then the v2 lift (Phase 3 — no tabs-exporter phase; see §6).** Classic-tabs integration is **ruled out** (outputs aggregate; per-respondent fitted probabilities are statistically inadmissible under D5 and redundant besides). CatDriver was not in V2_MIGRATION_PLAN's scope list; its landing zone is defined here: a tracking-style frozen `TR.CD` island + own workspace view, zero new row kinds, sequenced after keydriver's TR.KD to share scaffolding.

## 1. CRITICAL — silently wrong client-facing numbers or module-integrity breaks

### C1. Weighted multinomial importance is mathematically invalid — mixes weighted and unweighted log-likelihoods
The full multinomial model is fitted **with** weights (`04b_multinomial.R:98-102`, column `..catdriver_wt..`). The importance calculation refits reduced models from `model_result$analysis_data` — which is the raw data without any weight column (`04b:224`) — and passes **no weights** (`05_importance.R:213-214`); the LR statistic at `05:221` therefore subtracts an unweighted reduced log-likelihood from a weighted full one. The dead strip at `05:191-193` even looks for the wrong column name (`.wt`). **Coordinator-verified numerically:** with weights ~U(0.5,2), the as-coded "chi-square" for a real driver was **−282.1** (reader repro: −209.6) vs **+68.8** when the reduced model is correctly weighted. Negative chi-squares flow into `importance_pct = chi2/sum(chi2)` (`05:247-252`) → nonsense importance percentages, ranks, and p=1 significance in the headline Importance sheet, under PASS. Unweighted multinomial is unaffected.
**Fix:** carry the weight vector into `calculate_multinomial_importance` and every reduced-model (and null-model, H1) refit.

### C2. The subgroup-comparison feature is dead on arrival — column-contract mismatch, error swallowed under PASS
`build_or_comparison` reads `or_df$driver/$level/$or/$or_ci_lower/$or_ci_upper/$label` (`11_subgroup_comparison.R:265-292`). The production odds-ratios frame has **none of those columns** — it has `factor/comparison/odds_ratio/or_lower/or_upper/factor_label` (`09_mapper.R:333-435`; produced at `00_main.R:854`, no rename anywhere between). **Coordinator-verified by execution:** calling `build_or_comparison` with the real mapper schema throws `missing value where TRUE/FALSE needed` (NULL columns → `is.na(NULL)` → logical(0) in the `&&` at `11:287`). The call sits at `11:66` inside `build_subgroup_comparison`, so the error kills the **entire** comparison — importance matrix, Universal/Segment-Specific classification, model-fit table, auto-insights — which the `tryCatch` at `00_main.R:522-528` converts to a console `[WARNING]` and silently `NULL`. Excel sheets 9-11 then print "No subgroup comparison data - skipping" and the run ships PASS. Every subgroup-configured run since the schema change has silently lacked its subgroup comparison. The suite's 28 "Subgroup Comparison Logic" passes never construct an `odds_ratios` element at all (zero grep hits in `test_subgroup.R`), so the mocks skip the broken path.
**Fix:** align the consumer to the mapper schema (or rename at the boundary), add a real-schema fixture test, and make comparison failure degrade the run to PARTIAL with a degraded-reason instead of silent NULL.

### C3. The refusal system is broken module-wide — every refusal presents as BUG_INTERNAL_ERROR
Two incompatible definitions of `catdriver_refuse` exist: `08_guard.R:53` (live-intent, `reason=`/`fix=` — the signature used by ~87 call sites across the module) and `00_guard.R:42` (stale, `code=`/`how_to_fix=`, callers all dead). The GUI sources `08_guard.R` (`run_catdriver_gui.R:583`) then `00_main.R` (`:600`), whose **top-level code re-sources `00_guard.R`** (`00_main.R:49-59`) — so the stale signature clobbers the live one in every production session. **Coordinator-verified by execution:** after that source order, `catdriver_refuse(reason=..., fix=...)` throws `unused arguments (reason=..., fix=...)`, which the error handler (`08_guard.R:139-168`) formats as **"BUG_INTERNAL_ERROR: This is a bug. Report to the Turas development team"**. Reader B additionally proved the full GUI sequence live: loading the shipped template surfaces `CFG_SETTINGS_STRUCTURE_INVALID` as an internal bug with the fix text destroyed. The suite never catches this because the test helper sources `R/` files in *sorted* order (`helper-paths.R:58` — `00_guard.R` early, `08_guard.R` near-last), binding the **correct** signature in tests — the opposite of production (coordinator-verified by reading both source sequences).
**Fix:** delete the duplicate definitions in `00_guard.R` (see M8 — the whole file is dead code apart from this landmine) and add a regression test that a broken config yields a clean refusal with non-empty `how_to_fix`, sourced in GUI order.
