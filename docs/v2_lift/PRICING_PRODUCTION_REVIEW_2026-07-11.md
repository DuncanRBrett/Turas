## 0. Executive verdict

The happy-path math is sound: Van Westendorp intersections are delegated to the reputable `pricesensitivitymeter` package, Gabor-Granger demand/revenue/arc-elasticity formulas are correct, monadic logistic estimation is standard, and — unlike maxdiff/conjoint — the report layer already sits on shared TurasPins with a simulator whose JS math matches the R engine at the exported grid points.

But the module is **not weighted-study safe and not sequential-design safe**, and those are exactly the studies Turas runs. Three CRITICALs: on any weighted study the headline VW price points are silently **unweighted** while the stats pack prints "Weighting applied: Yes" (and GG/monadic *are* weighted, so methods disagree for a spurious reason); stop-early Gabor-Granger designs silently inflate demand at high prices and bias the recommended price upward; and the VW confidence-interval table is computed on a *different estimator and different population* than the headline points it claims to bracket. Below those: the shipped config template **cannot run either headline method** (VW/GG sheet names never normalised → cryptic non-TRS crashes), 1/2-coded Yes/No data becomes 100% purchase intent, and the report's own Save function destroys analyst commentary and added slides.

**Trust boundary today:** VW/GG numbers are trustworthy only for an unweighted study, GG in randomised full presentation with genuinely 0/1-coded responses and near-complete answers, and VW intransitive rates low enough (≲5%) that population mismatches don't move the numbers.

**Order of work: correctness first (Phase 1), then report persistence/escaping fixes (Phase 2), then tabs integration (Phase 3).** Phase 4 (a v2 report view) is **recommended against** for pricing — see §6.

## 1. CRITICAL — silently wrong client-facing numbers

### C1. Weighted studies get UNWEIGHTED Van Westendorp price points while the deliverable declares them weighted
`R/03_van_westendorp.R:516-534` — `psm_args` passed to `pricesensitivitymeter::psm_analysis()` contain no weights and no design object; the package's `psm_analysis_weighted()` exists in the installed version (1.3.3) *(reader-verified)* but is never called. Meanwhile GG demand is weighted (`04_gabor_granger.R:342-349`), the monadic glm is weighted (`13_monadic.R:125-127`), WTP extraction is weighted (`07_wtp_distribution.R:51-54`), the stats pack writes `weight_info[["Weighting applied"]] <- "Yes"` whenever a weight variable validates (`00_main.R:743-746`), and `TECHNIQUE_GUIDE.md:199` states "When survey weights are applied, the module uses them in all calculations."
**Consequence:** on a rim-weighted study, headline OPP/IDP/PMC/PME are unweighted while the GG and monadic optima are weighted — the methods disagree for a spurious reason and the client-facing audit trail asserts the opposite.
**Fix:** route VW through `psm_analysis_weighted()` (survey-design path), or refuse/prominently disclose "VW computed unweighted" whenever a weight variable is configured.

### C2. Gabor-Granger per-price denominators silently drop NAs — sequential (stop-early) designs yield inflated demand and an upward-biased optimal price
`R/04_gabor_granger.R:339` — each price rung's base is "whoever has a non-NA response at that rung": `gg_data[gg_data$price == p & !is.na(gg_data$response), ]`. The completeness exclusion in validation applies **only to VW** (`02_validation.R:445-453`); GG missingness only produces a warning line per column (`02_validation.R:425-436`). `docs/QUESTIONNAIRE_DESIGN_GUIDE.md:436-446` presents sequential presentation ("can stop early") as a supported design *(reader-verified)*.
**Consequence:** ascending ladder stops after the first "No" → respondents rejecting R40 are never asked R50 → NA → dropped → demand at R50 is computed only among survivors → the demand curve barely falls and `which.max(revenue_index)` lands at or near the top price. No warning is emitted when per-price bases differ.
**Fix:** refuse (or count + warn loudly) when per-price n varies materially; add an explicit "impute No after stop" option for sequential data.

### C3. The VW confidence-interval table brackets a different estimator and a different population than the headline price points
Headline run: `psm_analysis(..., validate = TRUE)`, unweighted (`03_van_westendorp.R:521`, C1). Bootstrap: `validate = FALSE` (`03:763`), **weighted** resampling probabilities when weights exist (`03:743-749`), and the table's `estimate` column is the *bootstrap mean*, not the reported point (`03:798-807`). Two readers independently proved `validate` moves the numbers (OPP 3 vs 4 on one synthetic set; IDP 11.19 vs 11.58 with 15% intransitives on another) *(reader-verified empirically; mechanism coordinator-verified at the cited lines)*.
**Consequence:** with intransitive respondents and/or weights, the CI table is centred off the headline points; a published price point can fall outside its own CI in the same workbook.
**Fix:** bootstrap the exact estimator reported — same `validate` flag, same weighting treatment — and report the actual point estimate in the `estimate` column.

## 2. HIGH

- **H1. The shipped template cannot run either headline method.** The template's VanWestendorp and GaborGranger sheets carry Title_Case setting names (`Col_Too_Cheap`, `Data_Format`, `Price_Sequence`, … — verified by reading the shipped `docs/templates/Pricing_Config_Template.xlsx` this session), but `load_van_westendorp_config`/`load_gabor_granger_config` build the list straight from the sheet with **no name normalisation** (`01_config.R:401-424, 432-466`) and every downstream read is lowercase — unlike Monadic and Validation, which have name maps (`01_config.R:591-607, 551-563`). The sheet-loader path is the one production takes whenever the sheets exist (`01_config.R:226, 235`). Reproduced crashes *(reader-verified)*: VW → "attempt to select less than one element in get1index" (`02_validation.R:356-361`); GG → "argument is of length zero" (`02_validation.R:425`, `if (NULL == "wide")`). Neither is a TRS refusal. **Fix:** add the same name-map treatment to the VW/GG loaders + a generate→load round-trip test.
- **H2. Numeric-list separators are inconsistent three ways.** Template instructs semicolons for GG prices ("e.g., 4.99;6.99;8.99", `lib/generate_config_templates.R:452`); the GaborGranger sheet loader splits on **comma** (`01_config.R:439, 444`); the Settings-sheet fallback path splits on **semicolon** (`01_config.R:510, 513`); USER_MANUAL says commas *(reader-verified)*; the price-ladder splits `tier_names` on semicolon (`11_price_ladder.R:65`). Following the template's own instruction yields an all-NA price sequence. Live proof: the main-pipeline test run refuses ladder generation with `CFG_TIER_NAMES_MISMATCH` "1 tier names provided but n_tiers is 3" (observed in this session's suite log) — and the suite still passes. **Fix:** one shared separator-tolerant parser; align template text.
- **H3. `VW_Monotonicity_Behavior = flag_only` (the default) claims violators are "retained" but they are silently excluded from the analysis.** `02_validation.R:403-409` prints "flagged but retained (behavior: flag_only)", yet the analysis hard-codes `validate = TRUE` (`03:521`), which excludes intransitive respondents inside the package (two readers proved the exclusion moves price points). Consequences: `flag_only` and `drop` produce identical estimates (the config knob is illusory); module diagnostics/n (`03:617-620`, stats pack `00_main.R:702`, segment table `10_segmentation.R:248`) overstate the analysed base; descriptives (`03:605-610`) include respondents the curves exclude. **Fix:** wire `validate` to the behavior setting, report psm's own analysed n / `invalid_cases`, correct the warning text, align the bootstrap (C3).
- **H4. Monadic logistic regression treats sampling weights as frequency weights.** Raw weights from the data (`13_monadic.R:82-87`, never normalised) go into `glm(..., weights = case_weights)` (`13:125-127`). Empirically *(reader-verified)*: population-scaled (grossing) weights make the fit degenerate (slope −5.9e13 → step demand curve, wrong optimum); even mean-1 weights overstate significance (no design-effect adjustment), feeding the "price effect statistically significant" gate (`13:148-151`) and synthesis claims (`12_recommendation_synthesis.R:394-414`). **Fix:** normalise to mean 1 at minimum + caveat the p-value, or `survey::svyglm`; refuse grossing-scale weights.
- **H5. "Binary" response coding maps any positive number to "buy" — 1/2-coded Yes/No data becomes 100% purchase intent at every price.** GG: `as.numeric(response > 0)` (`04_gabor_granger.R:244-246`); monadic identically (`13_monadic.R:77`). No validation anywhere that a declared-binary column is 0/1 (`02_validation.R:422-437` counts missing only). Alchemer exports 1=Yes/2=No routinely. Symptom is a flat 100% curve with optimum = max tested price; nothing flags it. **Fix:** refuse when a declared-binary column contains values outside {0,1} (offer explicit {1,2} mapping).
- **H6. Monadic observed cell intents are unweighted while the demand model is weighted.** `cell_intents <- tapply(intents, price_factor, mean)` (`13_monadic.R:104`) vs the weighted glm (`13:125-127`). Under weighting, the `Mon_Observed_Data` sheet is inconsistent with the fitted curve in the same workbook — and with a correctly weighted tabs table of the same question. **Fix:** weighted cell means.
- **H7. Stats pack "Effective N" is not an effective N.** `00_main.R:746` writes `ws$n_valid` (count of valid weight values) under the label "Effective N", while `TECHNIQUE_GUIDE.md:199-199` explicitly tells the statistician to check that figure for weighting-induced shrinkage. **Fix:** Kish `(Σw)²/Σw²` — from the shared consolidated helper once programme item OPUS-0 lands (until then, compute locally with a TODO pointing at OPUS-0); keep "Valid N" as its own row.
- **H8. `Generate_Stats_Pack = N` and the GUI stats-pack checkbox are both non-functional — the stats pack always generates.** `00_main.R:562-564` reads `config$settings$Generate_Stats_Pack`, but the loaded config **is** the flat settings list (`01_config.R:127-183`) — there is no `$settings` element, so the read is NULL → default "Y" → the `||` with the GUI option can never turn it off (`run_pricing_gui.R:188-190` *(reader-verified)*). A `*_stats_pack.xlsx` always lands next to the client deliverable. **Fix:** read the flat key; make the GUI option authoritative.
