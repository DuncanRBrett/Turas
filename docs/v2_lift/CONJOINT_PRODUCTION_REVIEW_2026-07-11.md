## 0. Executive verdict

The core aggregate path is genuinely sound: mlogit estimation, dfidx indexing, zero-centering, importance math, the client-side simulator engine, and the Excel simulator formulas were all checked and passed (§ "verified sound" notes below). Unlike maxdiff there is **no silent HB fallback** — bayesm failure refuses loudly — and no numeric-resp-id phantom-item analogue.

But the module is **not solid outside the happy path**: latent-class runs deliver an all-zero utilities table labelled PASS; the "simultaneous" best-worst estimator is statistically broken by construction; and every HB/LC run ships confidence intervals, p-values and significance stars computed from the wrong statistic (between-respondent heterogeneity SD masquerading as a standard error — found independently by three of the four readers). The shipped template's own dropdown values refuse the run for two settings. The GUI reports success on refusal. Several documented features (interactions, optimizer, custom images, Alchemer column mapping) are silently ignored or dead.

The HTML report is previous-generation architecture (server-side pre-rendered, hidden divs) — a dead end per `modules/tabs/docs/V2_MIGRATION_PLAN.md`, which already scopes conjoint. The embedded simulator is architecturally v2-like (JSON island + JS compute) and is the natural seed of the migration. **One correction to that plan:** it lists the Market Simulator as an already-standalone tool; in reality the standalone simulator was deprecated 2026-03-12 and merged into the combined report (`lib/html_simulator/99_simulator_main.R:36-71` is a forwarding wrapper; its sibling files are dead code).

**Order of work: correctness first (Phase 1), then report quick fixes (Phase 2), then the tabs exporter (Phase 3), then the v2 lift (Phase 4).**

## 1. CRITICAL — silently wrong client-facing numbers

### C1. Latent-class runs deliver an all-zero utilities table, silently, labelled PASS
`R/04_utilities.R:30` dispatches to HB extraction only when `method == "hierarchical_bayes"`; LC results carry `method = "latent_class"` (`13_latent_class.R:618-621`) so they fall into the aggregate branch. HB/LC coefficients are named `paste0(attr, "_", level)` (`11_hierarchical_bayes.R:139`), but the aggregate extractor strips only the attribute-name prefix (`04_utilities.R:113-121`), leaving `"_Level"`, which fails the `level %in% names(utilities)` check (`04_utilities.R:126`) — every utility stays at its initialised 0. *Empirically verified by executing the extraction logic in R this session (mlogit-style names match; `attr_level` names yield all zeros).*
**Consequence:** for `estimation_method = "latent_class"`, the Part-Worth Utilities sheet, Importance sheet, Market Simulator and HTML report (`07_output.R:86-96`, `00_main.R:446,463`) are built from an all-zero table; importance emits only a console `CONJ_ZERO_RANGE` info line (`04_utilities.R:204`) and the run reports PASS. The LC class sheets are populated separately from `model_result$latent_class`, so the workbook looks superficially plausible.
**Fix:** dispatch `latent_class` through `extract_hb_utilities` (the LC result carries all required fields: `attribute_map`, `col_names`, `individual_betas`); add a guard that refuses if all non-baseline utilities are exactly zero.

### C2. Best-worst "simultaneous" method estimates best and worst against each other — attenuated coefficients
`convert_best_worst_to_choice` (`10_best_worst.R:133-174`) stacks best-sets (chosen = best) and worst-sets (chosen = worst) with **no sign reversal of the design** — its own comments (:156-163) defer the reversal to "(best − worst)/2", which only the *sequential* path performs (`:291`). `estimate_best_worst_simultaneous` (`10_best_worst.R:321-350`) claims "choice_type as a covariate" but fits one standard attributes-only MNL (`build_mlogit_formula`, `03_estimation.R:399-416`) on the stacked data; `choice_type` never enters the formula.
**Consequence:** the same β must maximise P(best|β) *and* P(worst|β); the worst half pushes β toward −β, so estimates are biased toward zero. Output is labelled a successful `best_worst_simultaneous` model with no warning. (The sequential default is a defensible standard approximation and is fine.)
**Fix:** negate the worst-task design rows, or refuse the simultaneous method until properly implemented (recommended — see handover).

### C3. HB/LC "SE", CIs, p-values and significance stars are computed from the between-respondent heterogeneity SD
`aggregate_sds <- apply(individual_betas, 2, sd)` (`11_hierarchical_bayes.R:406`) is population heterogeneity, not the posterior SE of the mean — yet it is stored as `std_errors` (`:425`) and drives `SE`, `CI = β ± z·sd` and `p = 2(1−Φ(|β/sd|))` (`11_hierarchical_bayes.R:718-728`; same for LC at `13_latent_class.R:587-588`). Propagation verified: report "Std. Error" column (`lib/html_report/02_table_builder.R:56-81`), Excel Raw Coefficients z/p/significance-stars (`07_output.R:385-402`), and the utilities Interpretation column ("Not significantly different…", `04_utilities.R:34-40`).
**Consequence:** HB/LC intervals are inflated by roughly √n_respondents; client-facing significance claims are grossly conservative/wrong on every HB run. Direct analogue of maxdiff M5, but here on the module's **primary** estimation path, which is why it ranks CRITICAL. MNL runs are unaffected (real SEs).
**Fix:** compute population-mean uncertainty from the posterior draws of the mixture means (or at minimum sd/√n), and surface the heterogeneity SD separately, honestly labelled ("Heterogeneity (SD)"). LC class utilities additionally pass `std_errors = rep(0, …)` → p = 0 "significant" rows (`13_latent_class.R:698` → `11_hierarchical_bayes.R:728`); no production caller found (possibly dead API) but fix alongside.

**Phase 1 — Make it honest (blocks everything).** C1-C3, H1-H12, M1-M5 + §5 test additions. Everything is specified above with file:line. **Phase 2 — Current-report quick fixes.** P1-P8 (small, keeps live deliverables honest while the v2 lift lands; conjoint sits last in the default module order, so System A lives longest here).

**Phase 3 — Tabs integration (M effort, high value).** Reader-D design, verified against both sides of the socket:
- **Flagship payload: per-respondent attribute-importance shares as an Allocation question.** They sum to exactly 100 by construction (`11_hierarchical_bayes.R:816-821`) matching tabs' `{code}_1..{code}_k` contract (`allocation_processor.R:79`) — but the per-respondent matrix is currently computed and thrown away inside `calculate_attribute_importance_hb` (`:790-833`); a small engine change must expose it.
