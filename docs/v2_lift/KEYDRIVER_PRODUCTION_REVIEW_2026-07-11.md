## 0. Executive verdict

The module's headline number is right and its second billing is wrong: **Shapley-value R² decomposition — the column the importance table is sorted by — is implemented correctly** (exact all-subsets LMG with the proper kernel, weighted subset refits), but **"Johnson's Relative Weights" is mathematically broken** (C1): with two drivers it returns 50/50 *regardless of the data*, and with correlated drivers (i.e. always, in driver analysis) it is silently wrong, in all three code paths including the bootstrap CIs that appear to confirm it. Around the engine sit a broken segment-comparison layer (C2), a shipped config template that **cannot be loaded by the module's own loader** (H1, empirically proven), a GUI that prints "Analysis complete!" after a refusal (H2), and a provenance stamp asserting a primary method that is dead code (H3).

The v10.3/v10.4 feature tail (partial R², permutation, preflight validators, NCA, config-driven insights/slides) is substantially unwired or inert: documented, sometimes validated at config load, never executed or never rendered.

The HTML report is previous-generation architecture (server-side pre-rendered, R-baked SVG) but is genuinely on shared TurasPins already. Per the integration reader: **classic tabs integration is ruled out** (aggregate-only regression output, nothing per-respondent to crosstab), and the migration plan's "keydriver reuses the `index_scores` row pattern" premise is false — the correct v2 route is a tracking-style frozen island + own workspace view (§6 Phase 4).

**Order of work: correctness first (Phase 1), then config/UX/report honesty (Phase 2), then the v2 island lift (Phase 3, gated).** No tabs-exporter phase exists for this module.

## 1. CRITICAL — silently wrong client-facing numbers

### C1. "Johnson's Relative Weights" is mathematically wrong; with 2 drivers it ALWAYS returns 50/50 **[verified]**
`R/03_analysis.R:283-305` (main path), `R/03_analysis.R:699-707` (mixed path), `R/05_bootstrap.R:432-439` (bootstrap path — same math, so the CIs corroborate the wrong point estimates).

The code eigen-decomposes R_xx = VDV′ and then uses `Phi <- vecs %*% Lambda_sqrt` (= VD^½) as the loadings and `r_z_y <- Lambda_inv_sqrt %*% t(vecs) %*% r_xy` (= D^-½V′r) — i.e. it orthogonalises via **principal components**. Johnson (2000) requires the **symmetric square root** Λ = VD^½V′ and β* = Λ⁻¹r_xy; both rotations back by V′ are missing.

**Coordinator verification (independent script, this session):** two drivers with true effects 0.8 vs 0.1 (r₁₂=0.5) → code returns **50.0/50.0**; correct Johnson returns **80.8/19.2**. The correct raw weights sum exactly to model R² (0.4162); the code's sum to 0.562 (algebraically Σr²_zy, not R²). With exactly two predictors the eigenvectors of any 2×2 correlation matrix are (1,1)/√2 and (1,−1)/√2, so V²jk = ½ everywhere and the formula returns equal importance for both drivers *whatever the data*. The rescale at `03_analysis.R:308-311` (raw × R²/Σraw) and %-normalisation hide the sum discrepancy, so output looks plausible.

**Consequence:** the `Relative_Weight` column (always produced), `RelWeight_Rank`, `Average_Rank` (contaminated), the exec-summary method-agreement narrative, the bootstrap Relative_Weight CIs, and any quadrant run with `importance_source="relative_weights"` are all silently wrong whenever predictors are correlated. The workbook README sheet tells the client this is "Johnson's method" that "handles multicollinearity well" (`R/04_output.R:388-391`). Tests check only sum=100 and non-negativity (`tests/testthat/test_core_importance.R:112-136`) — any normalised vector passes.
**Fix:** Λ = `vecs %*% Lambda_sqrt %*% t(vecs)`; rw = Λ² %*% (Λ⁻¹ r_xy)². Apply identically in all three paths; drop the R²-rescale (correct raw weights already sum to R²); in the mixed path also use **weighted** correlations when a weight is configured (today `cor(mm)`/`cor(mm,y)` are unweighted, `03_analysis.R:676-677` **[verified]**, while the non-mixed path uses the weighted matrix). Golden test: hand-computed 2-driver case + k=3 case vs a reference implementation; assert raw RW sums to R² pre-normalisation.

### C2. The segment-comparison layer is wrong four ways at once, and its failures vanish under a PASS banner **[verified]**
All four defects coordinator-re-read this session:

- **(a) Unweighted under weighted studies.** Per-segment models are plain `lm` with no weights argument (`R/07_segment_comparison.R:511-513`); the headline model is weighted (`R/03_analysis.R:89-94`). Segment importance silently disagrees in weighting with the Total numbers, undisclosed.
- **(b) Categorical drivers scored 0%.** `coefs[drv]` at `07_segment_comparison.R:530-536` — a factor driver's coefficients are named `drvLevel`, so the lookup is NA → `return(0)`. Categorical drivers appear in segment tables at 0%, ranked last, classified "Low Priority", and fed into the plain-English insights — fabricated zeros.
- **(c) The Segments sheet's semantics are ignored.** Only `config$segments$segment_variable[1]` is used (`R/00_main.R:643`); rows 2+ and all `segment_values` groupings (documented at `docs/06_TEMPLATE_REFERENCE.md:555-570`, e.g. "customer_tier IN ('Gold','Platinum')") are discarded — `run_segment_importance_comparison` is called without `segment_values` (`00_main.R:646-652`), so it splits on every raw value of the first variable.
- **(d) Failures — including TRS refusals — are swallowed.** The whole step runs inside `tryCatch(error=...)` → `[WARN] Segment comparison failed` + NULL (`00_main.R:653-656`), with **no** `degraded_reasons` entry, so run status stays PASS. Refusal conditions inherit class `"error"` (`modules/shared/lib/trs_refusal.R:246` **[verified]**), so even `DATA_SEGMENT_VAR_NOT_FOUND` disappears into a console WARN. Compounding: a configured-but-absent segment variable is silently dropped at validation (`R/02_validation.R:116-117`, "optional") — a template-derived config typically hits exactly this.

**Consequence:** a user who defines five segments per the docs gets an analysis of one variable's raw levels — unweighted, with categorical drivers zeroed — or nothing at all, under a PASS status. Client-facing via the HTML segment section + insights.
**Fix:** weighted per-segment `lm`; aggregate factor terms per driver (reuse `build_term_mapping`) or exclude categoricals with an explicit note; iterate all Segments rows applying `segment_values`; segment failure → `degraded_reasons` → PARTIAL. Min-base is also hard-coded 30 (`00_main.R:651`), not the project's `significance_min_base` — align while in there.
