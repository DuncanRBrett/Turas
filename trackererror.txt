That debug log is the smoking gun — and the failure is inside run_tracker.R before it can even source lib/00_guard.R.Root cause (confirmed from the Tracker code)In modules/tracker/run_tracker.R you have:# When run from GUI, sys.frame(1)$ofile is NULL, so use getwd() as fallbackscript_dir <- tryCatch({  dirname(sys.frame(1)$ofile)}, error = function(e) {  getwd()})On macOS (and in Shiny), sys.frame(1)$ofile is often NULL. The problem is:* dirname(NULL) does not throw an error.* It returns character(0) (an empty-length character vector).* Then:source(file.path(script_dir, "lib", "00_guard.R"))becomes:* file.path(character(0), "lib", "00_guard.R") ? character(0)* source(character(0)) ? file(filename, "r", ...) : cannot open the connectionThat matches your log perfectly: readLines(run_script) works, then source(run_script) fails when it hits the first source(file.path(script_dir, ...)).Minimal fix (robust script_dir resolution)Patch run_tracker.R to explicitly handle NULL / length-0 / non-existent file:--- a/modules/tracker/run_tracker.R+++ b/modules/tracker/run_tracker.R@@-# Source module files-# When run from GUI, sys.frame(1)$ofile is NULL, so use getwd() as fallback-script_dir <- tryCatch({-  dirname(sys.frame(1)$ofile)-}, error = function(e) {-  getwd()-})+# Source module files+# In Shiny / GUI contexts sys.frame(1)$ofile is often NULL and dirname(NULL) returns character(0) (no error),+# so we must explicitly guard and fall back to getwd().+ofile <- sys.frame(1)$ofile+script_dir <- if (is.null(ofile) || length(ofile) == 0 || !nzchar(ofile) || !file.exists(ofile)) {+  getwd()+} else {+  dirname(ofile)+}That alone should stop the “cannot open the connection” on the first guard-layer source().Why this shows up only now after refactorThis “dirname(NULL) ? character(0)” trap is super common in refactors where you standardize “find script directory” logic.
